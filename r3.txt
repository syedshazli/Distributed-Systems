Q1. (Section 2) Raft is designed around the replicated state machine model. Briefly explain what this model is: what are the components, and what property must hold for the replication to be correct?

This model essentially relies on a log that contains the sequence of operations that should be executed by every server. The algorithms job is to make sure this sequence of operations is the same for everyone. In order for the replication to be correct, every log must contain the same requests in the same order. According to the paper, the main propert is in relation to safety, which states "if any server has applied a particular log entry to its state machine, then no other server may apply a different command for the same log index"

Q2. (Section 5.1) Raft uses terms to number successive leadership periods. What is a term, and how does it function as a logical clock? What does a server do when it receives an RPC with a term number higher than its own?

A term is just a contigous sequence of time for a leader to get work done. Terms function as logical clocks since a higher term number indicates an event happened later in time or after another event, thereby showing logical sequences of events. When a server receives an RPC with a term number higher than its own, it recognizes that server as the leader and accepts the RPC, making sure its new term number is now the same as the leaders term number.

Q3. (Section 5.2) When does a Raft server start a leader election, and what steps does it take? How does Raft guarantee at most one leader per term?

A Raft server starts a leader election when it receives no AppendEntries RPC response from a leader, and times out. It then starts an election by making itself a candidate, incrementing its own term number, and sends a RPC requests to other followers by asking for votes. Raft gurantees at most one leader per term because the candidate with the most votes is the only leader that serves a term. If there is a tie, then no leader is elected for that term.

Q4. (Sections 5.2 and 5.6) Raft uses randomized election timeouts to avoid split votes. Explain why randomization helps. The paper also states a timing requirement in Section 5.6:

broadcastTime « electionTimeout « MTBF

Explain in plain language what each inequality means and why it is necessary.

Randomization helps because if there was no randomized election timeout, it is possible that there is an infinite sequence of split votes, with no work actually being done since no leader is elected. Randomization ensures that one server times out early and broadcasts its request to be elected before other potential candidates get the chance, making those potential candidates followers instead. 

The election timeout cannot be greater than the average time to failure because if it was, then most servers won't have the chance to run since they will fail before even timing out and having the chance to be a candidate. The election timeout must also be greater than the broadcast time because we want to make sure that if a message is being sent by a real leader, then the followers can actually receive it, otherwise followers will think there is no response and try to start an election when a response is actually in transit.
