Q1. (Section 2.1) GFS relaxes one assumption that most previous file systems relied on. What is it, and what workload observation led to this design choice?

The assumption for consistency is relaxed. The workload observation that led to this was a burden on applications due to previously complicated file systems.

Q2. (Section 2.3) GFS separates two responsibilities that traditional file systems combine. What does the master handle, and what do chunkservers handle? Why is this separation useful?

The master maintains all filesystem metadata (current locations of chunks, mappings of files to chunks, etc). Chunkservers handle the actual storage of chunks on disk. This separation can be useful as the chunkserver can handle the process of replicating a chunk on multiple chunk servers, to ensure fault tolerance, while the master can know where things should be, while the chunkserver knows what's actually inside.

Q3. (Section 2.5) GFS uses a chunk size of 64 MB, much larger than a typical file system block. Give two advantages of this large chunk size.

One advantage of the large chunk size is the minimal client-master communication, since read and writes for the same chunk requires only one request to the master on chunk location. A second advantage is the reduced size of storage on the master

Q4. (Section 2.6) The master keeps two categories of metadata: non-volatile (persisted to disk) and volatile (kept only in RAM, reconstructed at startup). Identify which of the following is volatile and which is non-volatile, and briefly explain why each is stored the way it is:

File-to-chunk-handle mapping: This data is non-volatile. The file to chunk mapping is stored in the operation log. This is because the files to chunk mappings in log form serve as a timeline for concurrent operations. If this data was volatile, we risk having invalid copies of the locations on the filesystem in general.

Chunk version number: The chunk version number is non-volatile. The reason for this is because if the master crashes, it needs to know which copy is most recently updated

Which chunkservers hold a replica of a given chunk: This data is volatile. This is because we can just query the data from the chunkserver

The identity of the current primary for a chunk: This data is volatile because the identity of the primary can changhe often, so storing it to ram often can actually be very slow.

Q5. (Section 2.6.3) The master uses an operation log and checkpoints for crash recovery. Explain what each one stores, and describe the recovery procedure when the master restarts.

The operation log is a record of every metadata changing operation, which is replicated on multiple remote machines. When the master restarts, it simply replays the operation log, since it contains the record of metadata changes.

Q6. (Section 3.1) Describe the steps a GFS client takes to read a file, starting from having only a filename and byte offset. Your answer should include what the client sends to the master, what the master returns, and what the client does next.

First, the client send the filename and the chunk index, which is calculated using the byte offset. The master replies with a chunk handle, and where the replicas are stored. The client then sends the chunk handle and byte range to the nearest replicam and the chunkserver returns the data to the client.

Q7. (Section 3.1 and 3.3) GFS uses leases to designate one replica as the primary for a chunk. What problem do leases solve? What happens to the primary's authority when its lease expires?

Leases ensure a consistent mutation order, which solves the problem of differing mutation ordering. A replica that has the lease is known as the primary. The primary is no longer allowed to mutate the file once its lease expires.

Q9. (Section 2.7 and Table 1) GFS provides a relaxed consistency model rather than strong consistency. After a successful record append, what does GFS guarantee about the written region? What does it not guarantee about the regions between successful appends?

GFS gurantees the appended output will be preserved. It does not gurantee the regions between succesful appends can be preserved, as in component failiures can overwrite data.
